# Python Project Rules for Skills

Use this reference when a skill contains Python scripts, templates generated by Python, data processing code, browser automation, document rendering, or any other Python runtime behavior.

## What To Add

Add `pyproject.toml` when the skill has Python code that may be executed by another user or agent. The file should include:

```toml
[project]
name = "skill-name"
version = "0.1.0"
description = "Python runtime helpers for the skill."
requires-python = ">=3.11"
dependencies = []

[dependency-groups]
dev = [
    "pytest>=8.0",
    "ruff>=0.6",
]

[tool.ruff]
line-length = 100
target-version = "py311"
```

Keep runtime dependencies in `project.dependencies`. Put test and lint tools in the `dev` group.

Commit `uv.lock`. For skills, the lock file is part of portability.

## Format Layering

Validate the skill in two layers:

1. Agent Skills spec: `SKILL.md`, frontmatter, directory name, optional metadata, and progressive-disclosure structure.
2. Python engineering rules: `pyproject.toml`, `uv.lock`, package dirs, check scripts, tests, and runtime notes.

Run:

```bash
uv run python scripts/validate_skill.py /path/to/skill --python
```

Do not loosen the Agent Skills spec to fit Python packaging. If the repo is a single Python-backed skill, prefer making the repo root the skill root.

## Validation Command

Every Python-backed skill should have one cheap command that proves the runtime works:

```bash
uv run python scripts/check.py
```

The check should cover the smallest useful path:

- import every local script module that the skill tells Codex to use
- confirm required assets or fixtures exist
- run one tiny transformation or render
- fail with clear messages when an external binary, environment variable, or permission is missing

Avoid checks that need private credentials, large downloads, or fragile network calls.

## Tests

Add `tests/` when behavior can break quietly, especially parsing, file generation, data transforms, document rendering, browser automation, or command-line argument handling.

Prefer small fixture-based tests. Avoid tests that require the user’s home directory, system-specific paths, private credentials, or live services.

## Runtime Notes

Make runtime assumptions explicit in `SKILL.md` or a reference file:

- Python version
- fixed commands
- input and output paths
- writable directories
- environment variables
- network requirements
- browser, font, image, PDF, Office, or system binary requirements
- where generated files should be written
- what to do when a dependency cannot be installed in the sandbox

## External Resources

Bundle small resources inside the skill directory. For large resources, document:

- source URL
- expected destination
- checksum or another verification method
- whether network access is required
- fallback behavior when the resource is unavailable

## When Not To Add More

Do not add Python packaging ceremony when the skill only contains `SKILL.md` and references.

Do not add `README.md`, install guides, changelogs, or duplicate docs. Skill instructions should stay inside `SKILL.md` and focused references.

Do not create a large framework around a simple helper script. One `pyproject.toml`, one lock file, and one check command are often enough.
